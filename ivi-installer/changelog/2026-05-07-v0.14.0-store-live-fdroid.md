# 0.14.0 - Store tab is now backed by the live F-Droid index

The Store tab no longer ships a hand-curated 36-app list. It pulls the
full F-Droid catalog (4000+ apps) on first launch, caches it for 24 h,
and merges in a small bundled `extras.json` for the entries F-Droid
can't possibly cover (Telegram via direct download + 3 AppGallery
apps).

## What changed

- **Live catalog.** New helper `catalog.from_fdroid_index(index)`
  converts the parsed `index-v2.json` into our `CatalogEntry` shape
  (highest versionCode wins per package; localized name / summary
  picked with priority `ru → en-US → en → first-available`; F-Droid
  category tags mapped to our nine-category taxonomy via
  `_FDROID_CATEGORY_MAP`).
- **Background fetch.** `workers.CatalogFetchWorker` runs the index
  fetch + conversion off the UI thread. The Store tab seeds with
  bundled extras so AppGallery/direct rows render immediately, then
  the F-Droid entries flow in via `set_catalog` when the worker
  finishes. First launch ~3-10 s on a cold network; subsequent
  launches reuse the on-disk cache and finish in well under a second.
- **Refresh button** now bypasses the disk cache (force-refetches
  the index) instead of re-reading a static file.
- **Lazy rendering.** The Store tab renders the first
  `BATCH_SIZE = 80` `AppRow` widgets up front and appends another
  batch when the scroll bar passes 85 % of the way to the bottom.
  Building all ~4000 rows up front took multiple seconds and burned
  ~50 MB of widget overhead; with paging the tab opens instantly and
  rows materialise as the user scrolls.
- **Filter changes** detach the rendered window and start a fresh
  page from the top of the new filter result. Cached `AppRow`
  widgets are kept in `_rows` and reused when the user clears a
  filter - no rebuild cost.
- **Icons.** F-Droid icons load lazily from the live URLs picked off
  `metadata.icon.<lang>.name`. Bundled icons under
  `resources/store-icons/<id>.png` still win when present, so the
  most-used apps paint instantly without a network round-trip.
- **Gzip on the wire.** `fdroid.fetch_index` now sends
  `Accept-Encoding: gzip` and decompresses on the client. The
  index transfers as ~15 MB instead of ~50 MB raw.

## Files

- `catalog.py` - `from_fdroid_index`, `merge`, `load_extras`
  (renamed from `load`, with `load` kept as a back-compat alias).
- `resources/extras.json` - replaces `catalog.json`. Holds only the
  4 non-F-Droid entries (Telegram, Радиопоток, Кинопоиск, Я.Музыка).
- `ui/store_tab.py` - lazy-render plumbing: `_render_more`,
  `_on_scroll_changed`, `_filtered_ids` / `_rendered_ids`.
- `ui/workers.py` - `CatalogFetchWorker`.
- `ui/main_window.py` - `_kick_catalog_fetch`, hooked from
  `_build_store_tab` and `_on_store_refresh`.
- `sources/fdroid.py` - gzip transport.

## Tests

`test_catalog.py` rewritten around `from_fdroid_index` with synthetic
v2-shape fixtures (no real F-Droid network). New `test_store_tab.py`
cases cover lazy rendering: a 245-entry catalog renders exactly
`BATCH_SIZE` rows on init; `set_catalog` swap clears cached rows.
287 passed.

Version: 0.13.0 → 0.14.0 (minor - new behaviour users will notice).
