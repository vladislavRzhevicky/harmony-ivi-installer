# v0.8.6 - per-screen checkbox grid (Driver / Passenger / Rear)

The "Install on" section is no longer a binary "all vs driver" radio
pair - users can now pick any combination of cabin screens.

## What changed

- **3-card checkbox grid**: Driver, Passenger, Rear, all checked by
  default. Compact layout (no second-line description) keeps three
  cards comfortably side-by-side.
- **Dynamic mapping** (`devices.categorize_screens`): on device
  connect we run `pm list users` and bucket the multimedia user ids
  (10-13) into Driver/Passenger/Rear by parsing each user's name
  from the device. Falls back to a hardcoded layout
  (Driver=13, Passenger=11, Rear=10/12) when the probe fails or the
  names are unclassifiable. User id 13 is always treated as Driver.
- **Multi-user install path**: `InstallContext.target_users:
  tuple[int, ...] | None` is the new canonical knob; the legacy
  `target_user` is still accepted for backward compatibility. When
  multiple categories are selected, the install seeds via user 0
  (system surface, the verified install path) and fans out to every
  selected screen via `pm install-existing`. When a single screen is
  selected, the install goes straight to that user with no fan-out.
- **Pipeline filter** updated: the `Mirror to passenger screens` row
  hides when the selected target resolves to a single user (no
  fan-out to do), shows otherwise. Same logic as before - only the
  predicate changed from "target_user is not None" to "len(targets)
  == 1".
- **Persistence**: selected categories survive across sessions via
  the `install_target_screens` setting.
- **Validation**: install button is disabled with helper copy "Pick
  at least one screen." when no checkboxes are ticked.

## Why dynamic mapping over hardcode

User-id-to-physical-screen mapping varies by car model and firmware.
Huawei firmwares typically name their per-screen users meaningfully
("Driver", "Passenger", "Rear" or close variants) - pulling those
labels keeps the cabin model honest across different vehicles
without requiring a per-model code change. The hardcode kicks in
only when the probe returns nothing or names are uninterpretable.

## Compatibility

- `install_cascade` and `install_with_strategy` accept both
  `target_user` (single int, legacy) and `target_users` (sequence,
  new). When both are set, `target_users` wins.
- Public `Pipeline.set_stages` / `mark_*` API unchanged.
- `_selected_target_user()` helper in MainWindow renamed to
  `_selected_target_users()`. Old test assertions updated.
