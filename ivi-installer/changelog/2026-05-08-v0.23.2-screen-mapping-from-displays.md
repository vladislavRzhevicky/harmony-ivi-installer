# 0.23.2 - Driver/Passenger/Rear mapping from physical displays

## The bug

Tester reported that on his Deepal S09 the **Passenger** and **Rear**
checkboxes in the install grid were swapped - picking "Passenger"
actually targeted the rear screen and vice-versa. Confirmed live:

| displayId | display name | active user (real) | what we said |
|---|---|---:|---:|
| 0  | `control_panel`       | 13 | Driver âś“   |
| 3  | `hud_panel`           | 13 | (HUD)      |
| 4  | `central_rear_panel`  | 12 | Passenger âś— (should be Rear) |
| 6  | `co-driver_panel`     | 11 | Rear âś—     (should be Passenger) |

Root cause: `categorize_screens` resolved Driver/Passenger/Rear by
parsing **user names** from `pm list users`. On this firmware the
names are opaque (`NoLoginUser`, `NoLoginUser_4`, `-1661017452`),
heuristics didn't fire, and the categoriser fell back to a hardcoded
`_FALLBACK_CATEGORIES` layout (`passenger=(10,12), rear=(11,)`) that
just happened to be the **mirror image** of this car's wiring.

## Fix

Stop guessing from user names. Resolve roles from the **physical
display layer** instead - display names are fixed by the firmware
across Huawei IVI builds:

* `control_panel`       â†’ Driver
* `co-driver_panel`     â†’ Passenger
* `central_rear_panel`  â†’ Rear
* `hud_panel`           â†’ folded into Driver (same display group)

For each display, `dumpsys window displays` tells us which Android
user owns the focused window - that's the user id we put under the
matching role.

## Implementation

New in `ivi_installer/devices.py`:

* `DisplayInfo(display_id, display_name, user_id, role)` dataclass.
* `parse_display_names(text)` - extracts `{display_id: name}` from
  `dumpsys display` output (the `displayId Nname"` token).
* `parse_focused_users(text)` - extracts `{display_id: user_id}` from
  `dumpsys window displays`. Prefers `mCurrentFocus=Window{u<N>}`,
  falls back to the first `ActivityRecord{u<N>}` in that block when
  the display has no focus but holds a rooted task (e.g. screen OFF).
* `_classify_display_name(name)` - pattern table mapping name
  fragments to `driver` / `passenger` / `rear` / `hud`. Order is
  significant: more specific patterns first (`co-driver` before
  `driver`, `central_rear` before `rear`).
* `build_displays(names, focused)` - combines the two maps into a
  sorted `list[DisplayInfo]`.
* `list_displays(serial)` - runs the two `dumpsys` calls and returns
  the populated list, `[]` on adb errors.

`categorize_screens(users, displays=None)` now has a new resolution
order:

1. Display-based mapping (preferred), if `displays` was supplied and
   at least the driver display resolved cleanly.
2. User-name heuristics (existing behaviour) when displays are absent
   or didn't surface a driver.
3. `_FALLBACK_CATEGORIES` (last resort).

`UsersListWorker` now also calls `list_displays(serial)` and emits
`(serial, users, displays)`. `_on_screen_categories_loaded` plumbs
`displays=` through to `categorize_screens`.

## Why this is more robust

* Display names ship with the firmware and don't depend on whether
  `pm list users` returns "Driver" or `NoLoginUser_4`. Same probe
  works on cars where role names were never set.
* `dumpsys window displays` reports the live owner of each display,
  so we don't have to assume which Android user id corresponds to
  which seat - we read it.
* HUD (separate physical display, same display group as the driver)
  is correctly folded into Driver instead of becoming a phantom
  fourth bucket.
* The S09 wiring (passenger=11, rear=12) is now derived, not assumed
  - different cars / future facelifts will be picked up automatically.
