# 0.16.0 - Store catalog moves to a SQLite-backed query layer

The Store tab no longer holds the full F-Droid catalog (4335 entries)
in memory as a tuple of `CatalogEntry` dataclasses. Instead, the
catalog lives in a SQLite database; the tab queries it for the
filtered + sorted id sequence and only materialises `CatalogEntry`
objects for the rows currently visible on screen.

## Memory before / after (4335 entries)

| Step                                            | RSS Δ before     | RSS Δ after      |
|-------------------------------------------------|------------------|------------------|
| F-Droid index parsed and held in memory          | ~578 MB peak     | ~3.2 MB on disk  |
| Catalog data alive after parse                   | ~160 MB          | ~0.4 MB (page)   |
| Cold-start parse cost                            | ~280 ms (every launch) | ~0.2 ms (open existing) |

The 50 MB JSON cache on disk stays - it's the wire format. SQLite is
the query layer built from it.

## Architecture

- New module `ivi_installer/catalog_store.py` - `CatalogStore` class
  wrapping a `sqlite3.Connection`.
  - `from_apps(...)` - in-memory store for tests + extras-only seed.
  - `build_to_path(path, *, apps, generated_at)` - atomic on-disk
    rebuild inside an `IMMEDIATE` transaction. Writer uses WAL so
    the previous reader keeps working until COMMIT lands.
  - `open_readonly(path)` - UI thread's reader handle.
  - `query_ids(...)` and `count(...)` - pure SQL across all chip /
    sort / search / category / source filters; supports
    `LIMIT/OFFSET` for pagination.
  - `fetch(id)` and `fetch_many(ids)` - batch row → CatalogEntry.
- New schema: one `apps` table with pre-computed `name_lower` and
  `search_blob` columns (lower-cased concat for case-insensitive
  LIKE), B-tree indexes on category / source_kind / tested /
  added_at / last_updated_at / size_mb / name_lower. SQL plan stays
  on indexes for every filter combination.
- `merge_apps(primary, extras)` lives next to `CatalogStore` -
  extras win on id collisions.

## UI plumbing

- `StoreTab.__init__` now accepts `Catalog | CatalogStore | None`.
  Anything that isn't already a store gets wrapped via
  `CatalogStore.from_apps` so the rest of the tab only ever talks
  SQL. The internal `_entries` tuple is gone.
- `_apply_filters` calls `store.query_ids(...)` once per filter
  change to get the ordered id list.
- `_render_more` fetches only the ids it's about to display via
  `store.fetch_many(ids_window)` - no more touching 4000+
  `CatalogEntry` objects to find the 80 we need.
- `_ensure_row` (used by `set_install_phase` for off-screen rows)
  fetches by id on demand.
- `set_catalog(...)` swaps stores and closes the previous one to
  avoid leaking an open SQLite handle.

## Worker change

- `CatalogFetchWorker.result` now emits the SQLite path (str)
  instead of an in-memory `Catalog`. The worker:
  1. Pulls the F-Droid index (gzipped over the wire, 24 h cached).
  2. Parses to `tuple[CatalogEntry, ...]` and merges extras.
  3. Writes the SQLite at `~/.ivi-installer/cache/catalog.sqlite3`
     (atomic via in-transaction `DELETE` + `INSERT` on WAL).
  4. Emits the path; UI re-opens read-only and calls `set_catalog`.

## Warm start

`_build_store_tab` opens the cached SQLite if it exists and skips
the background fetch - every launch after the first populates the
tab instantly with the full ~4335 entries. Cold start (no SQLite)
seeds with the 4 bundled extras and kicks the live fetch in the
background, exactly like before.

## Tests

19 new `test_catalog_store.py` cases covering: round-trips, all
filter combinations, sort fallbacks (NULLs to the end), pagination,
on-disk build idempotency, read-only enforcement, extras-overlay
precedence. Existing 298 tests still pass; total **317 passed**.

Version: 0.15.0 → 0.16.0 (minor - internal architecture change,
visible only as faster startup and lower memory).
