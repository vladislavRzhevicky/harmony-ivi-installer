# 0.20.0 - Catalog shows apps in AppGallery's own order

The Sort dropdown is gone. The catalog now displays apps in the
exact order AppGallery returns them - categories iterate in
``DEFAULT_CATEGORY_URIS`` order (cars → navigation → travel →
entertainment → … → games → most_ranked), items inside each category
in AppGallery's own rank order. Curated extras (Telegram +
AppGallery picks) ship at the very top.

## Why

The custom sort options ("Name A-Z", "Recently updated", "Size") were
solving a problem that didn't exist for this catalog - AppGallery
already orders its app lists by relevance/popularity, and most of
the items don't have ``last_updated_at`` / ``added_at`` populated
anyway (those came from the F-Droid index that's no longer pulled).
Rendering apps in AG's own order is what users expect when they
think "open AppGallery's Cars list".

## Code

- `catalog.py` - dropped `SORT_OPTIONS` and `sort_apps()`.
  `from_appgallery_index` no longer alphabetises at the end; the
  dict-of-CatalogEntry preserves insertion order.
- `catalog_store.py` - `query_ids` always uses ``ORDER BY rowid`` and
  no longer accepts `sort_key`. `_build_order_by` removed.
  `merge_apps` now emits **extras first** so they take the lowest
  rowids and surface at the top of the All view.
- `ui/store_tab.py` - Sort combo + `_sort_key` state removed from
  the toolbar. `_reset_filters` no longer touches a sort combo.

## Tests

5 sort tests dropped. New tests:

- `test_query_ids_preserves_insertion_order` - verifies rowid order
- `test_filter_preserves_appgallery_insertion_order` - UI-level
  check that the rendered list matches input order, not alphabet
- `test_reset_filters_button_clears_state` rewired to use
  `category_combo` instead of the deleted `sort_combo`

294 passed.
