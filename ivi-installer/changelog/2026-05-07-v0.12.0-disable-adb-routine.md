# 0.12.0 - Disable ADB button (UDS stopRoutine)

## What changed

The Enable ADB tab gets a second button - **Disable ADB** - that
sends the symmetric UDS reverse of Enable: same DoIP/UDS sequence
through stages 1-5, then `31 02 03 0C` (RoutineControl
**stopRoutine** on 0x030C) at stage 6 instead of `31 01 03 0C`
(startRoutine).

This is the safest plausible reverse of Enable per ISO 14229:
sub-function 0x02 of RoutineControl is defined as "stop the routine
that was started with sub 0x01". If the ECU doesn't honour this
contract on the dealer routine - Avatr's binary doesn't ship a
disable function, so we have no confirmation it works - the worst
case is a clean negative response (NRC 0x12 / 0x24 / 0x31) which
gets surfaced verbatim in the pipeline. Nothing destructive, no
risk of corrupting state.

## Why "safest"

Three options were on the table for closing the port:

1. **`31 02 03 0C` (stopRoutine, this patch)** - same routine ID,
   sub-function 0x02. ISO-defined as the reverse of startRoutine.
2. `31 01 03 0D` (next routine ID) - pure guess at a paired ID.
   Could land on an unrelated routine.
3. `31 01 03 0C 00` (startRoutine with a parameter) - guess that
   the routine takes a 0/1 toggle byte.

Option 1 is the only one with explicit ISO backing and the only one
that the ECU is required by spec to either honour or reject
cleanly. Options 2 and 3 are educated guesses that could be misread
as another command. We ship #1.

## Implementation

`diag.py`:
- Refactored `enable_adb` into `_run_adb_routine(action, ...)` that
  takes `"enable"` or `"disable"`. Stages 1-5 unchanged. Stage 6
  swaps the UDS sub-function and the routine label, and relaxes the
  status-byte check for disable (we accept any positive 0x71
  response since stopRoutine semantics on this routine aren't
  documented).
- Public `enable_adb` and `disable_adb` are now thin wrappers around
  `_run_adb_routine`. Existing callers (tests, workers) keep their
  API.
- New `stages_for(action)` returns the right pipeline tuple - the
  trailing label changes between "Enable ADB" and "Disable ADB
  (stopRoutine)" so the operator sees what's running.

`ui/workers.py`:
- `EnableAdbWorker` now takes an `action` kwarg and dispatches to
  the right `diag.*_adb` function. No new worker class needed.

`ui/main_window.py`:
- New `Disable ADB` secondary button next to the primary Enable CTA.
- `_on_enable_adb` / `_on_disable_adb` are thin shims around a new
  `_run_diag_action(action)` that handles both. Pipeline labels
  rebuild via `set_stages(diag.stages_for(action))` before each run.
- Tooltip on the Disable button explains it's a `stopRoutine` and
  what happens if the ECU doesn't support it.

`tests/test_diag.py`:
- 3 new tests: `stages_for` round-trip, unknown-action rejection,
  signature parity between `enable_adb` and `disable_adb`.
- 16/16 diag tests pass; full suite at 231 passed.

## What this still doesn't fix

We don't know yet whether ADB-enable on this ECU is persistent
(survives reboot) or volatile. Once we have a successful enable on
the bench, the planned next step is a 30-second probe via shell:

```
adb shell "getprop | grep -iE 'adb|debug'"
adb shell ls -la /persist/property/ /data/property/
adb shell "logcat -d -t 200 | grep -iE 'adb.*enable|0x030C'"
```

If persistent - Disable ADB matters for cleaning up after a
service. If volatile - a reboot does the same job and Disable is
nice-to-have.
