# 0.10.1 - drop libc++.dylib, ship +x adb

Fixes two issues with the v0.10.0 macOS bundling.

## libc++.dylib was dead weight

The 0.10.0 changelog claimed `adb` on macOS loads
`@executable_path/lib64/libc++.dylib` and the `lib64/` subdir had to
travel with it. Wrong. Verified with `otool -L` and a smoke-test:

- `adb` only LC_LOAD_DYLIBs `libSystem`, `CoreFoundation`, `libobjc`,
  `IOKit`, `Security` - all on the system.
- Copied the bundled `adb` to a scratch dir with no `lib64/` next to
  it, ran `adb version` → exit 0.

The leftover `@loader_path/lib64` rpaths in the Mach-O are for
Google's *other* platform-tools (aapt2 etc.), not adb itself.

→ Removed `lib64/libc++.dylib` (2.6 MB), removed its `force-include`
entry, removed the `lib64/*` glob, fixed the docstring in
`_bundled_adb()` and the comment in `pyproject.toml`.

## adb in the repo was 0o644

The source file at `resources/platform-tools/darwin/adb` was committed
with mode 100644 - not a hatchling/NTFS artifact, just a missing
`chmod +x` before the initial commit. Consequence: wheel ships
non-executable adb, briefcase unpacks non-executable adb, and the
runtime fallback chmod in `_bundled_adb()` was the *only* thing
making the macOS build work.

→ `git update-index --chmod=+x` so the file is tracked as 100755.
The runtime chmod stays as a safety net for any pipeline that
strips +x (e.g. zip/tarball roundtrip on Windows), with the
docstring updated to reflect that it's a fallback, not the primary
mechanism.

## .app size impact

−2.6 MB unpacked, slightly less in the .dmg.

## Files touched

- `ivi_installer/__init__.py` - 0.10.0 → 0.10.1
- `pyproject.toml` - version, drop libc++ from `force-include` and
  the `lib64/*` glob, fix the bundle comment
- `ivi_installer/adb.py` - corrected `_bundled_adb()` docstring
- `ivi_installer/resources/platform-tools/darwin/adb` - mode
  100644 → 100755
- `ivi_installer/resources/platform-tools/darwin/lib64/libc++.dylib`
  - deleted
