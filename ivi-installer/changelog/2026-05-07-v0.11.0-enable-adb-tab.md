# 0.11.0 - Enable ADB over Ethernet (DoIP/UDS)

## Why

After the latest IVI firmware push, ADB over USB stops responding on
Avatr/Deepal head units that come from the factory locked. The only
remaining channel into the box is the diagnostic Ethernet bus on the
OBD port (192.168.69.0/24). The dealer toolchain (and the competing
`AvatrAppInstaller.exe` from `/AVATR/`) flips a SecurityAccess-gated
RoutineControl on ECU 0x0300 to bring the USB ADB endpoint back. We
ported that sequence into the installer so the tech doesn't have to
keep a Windows VM around just to unlock the box.

## What this ships

A new top-level tab - **Enable ADB** - with a single primary CTA. The
operator plugs into OBD, sets the laptop IP in 192.168.69.x, ignition
ON, hits the button, and watches a six-stage pipeline:

1. mTLS handshake on the TLS gateway:30504
2. DoIP TCP connect on the DoIP gateway:13400
3. Routing activation
4. Diagnostic session (UDS 0x10 0x03)
5. Security access (UDS 0x27, seed → key, key → ack)
6. RoutineControl 0x030C (UDS 0x31 0x01) - Enable ADB

Defaults: DoIP gateway `192.168.69.6`, TLS gateway `192.168.69.21`
(both editable, persisted in `~/.ivi-installer/settings.json`). A
"Discover" button broadcasts a DoIP VehicleIdentificationRequest and
auto-fills the DoIP IP from the first responder, useful when the IVI
image has different gateway addresses.

## How it works

`ivi_installer/diag.py` is the protocol port. Pure stdlib - `socket`,
`ssl`, `struct`, `tempfile` - no extra deps. Reproduces, byte-for-byte:

- ISO 13400 DoIP frames (header `02 FD <ptype:H> <len:I>`).
- ISO 14229 UDS dialog: DiagSession → SecurityAccess → RoutineControl.
- The `Seed2Key` algorithm reverse-engineered from the AvatrAppInstaller
  PyInstaller binary (mask = `[0x34, 0x12, 0x3E, 0xC6]`, with a bit-
  reversal pass and a 32-bit modular sum). Locked-in test vectors in
  `tests/test_diag.py`.
- The Tester certificate chain + Ed25519 private key embedded as PEM
  constants. mTLS context built with `CERT_NONE` + legacy server
  connect (Avatr's gateway runs an old TLS stack).

The UI side is a new `_build_enable_adb_tab` in `main_window.py` plus
two new workers in `ui/workers.py` (`EnableAdbWorker`,
`DiscoverGatewaysWorker`) - both backed by `run_in_thread` so the Qt
event loop stays responsive while the diagnostic exchange runs.

## What this does NOT do

- Doesn't touch USB ADB itself - once the routine succeeds, the IVI's
  USB ADB endpoint comes back online via normal cable. The existing
  device-poller in the app picks it up.
- Doesn't claim to work on every IVI variant. The ECU layout was
  observed on 2024+ Avatr/Deepal builds. If a customer's gateway lives
  on a different IP, "Discover" picks it up; if the ECU layout
  differs, the failure surfaces in the pipeline at the offending
  stage with a UDS NRC byte for diagnosis.
- Doesn't run a continuous TLS-keepalive thread. The mTLS socket is
  held open just long enough to keep the gateway awake during the
  ~few-second UDS handshake and then closed. If the gateway falls
  asleep before the routine completes (very slow link), retry.
