# 0.23.1 - Store install: stop the scroll-jump on click + verbose breadcrumb

Tester reported: clicking Install on a row in the AppGallery list
**scrolled the catalog to the middle of the list** with no progress
bar, button stayed "Install" (after manually scrolling back), and a
second click toasted "wait for the current operation to finish".

## Root cause for the scroll-jump

`QPushButton` defaults to `Qt.StrongFocus` - it accepts both click
and tab focus. So when the user clicks Install:

1. The button gains keyboard focus.
2. `_on_store_install` runs `_set_busy(True)` →
   `set_global_busy(True)` → `_refresh_install_buttons(force_disable=True)`.
3. That iterates every row's button and calls `setEnabled(False)`.
4. Disabling the **focused** widget makes Qt move focus to the next
   focusable widget in the tab chain. With every Install button
   disabled, the next focusable widget is some unrelated button
   several rows down.
5. `QScrollArea` auto-scrolls to keep the focused widget visible →
   the catalog jumps "to the middle".

Fix: each row's Install button now declares `Qt.NoFocus`. The button
is mouse-only - clicks still work, hover styling still works, but
Qt's focus chain skips it entirely. Disabling it can no longer yank
focus elsewhere, so no scroll-jump.

## Why the button stays "Install" after scrolling back is also fixed

Same root cause, knock-on effect. With focus correctly staying put,
the row that received the click gets its `set_install_phase(...,
PHASE_RESOLVING)` call processed normally and the button re-paints
to "Resolving…" → "Downloading NN%" → "Installing…" without the user
having to scroll back to find it.

## Verbose breadcrumb in `_on_store_install`

Three new `_log` lines surface what happened inside the click
handler in the in-app log panel:

- `store: install click - id=…  name=…  sources=[…]`
- `store: install rejected - _busy=True` (when the busy guard fires)
- `store: install rejected - no head-unit serial` (when the device
  guard fires)

Both rejection paths used to fail with just a toast; the toast
disappears, the user is left guessing. Now whoever shares logs can
see the click registered and which guard tripped.

## Files

- `ivi_installer/ui/store_tab.py` - `setFocusPolicy(Qt.NoFocus)` on
  each `AppRow._button`.
- `ivi_installer/ui/main_window.py` - three `_log` lines at the top
  of `_on_store_install`.
- `__init__.py`, `pyproject.toml` - version 0.23.1.

## Tests

309 still pass. Existing store-tab tests don't probe focus policy
directly; the fix is observable as a one-liner declaration on a
member widget.
