# 0.22.2 - Trial-build expiration check

Primitive doormat lock for hand-out test builds: at startup, if the
current date is more than 14 days past `_BUILD_DATE`, the program shows
a single message and exits with code 2 instead of launching the GUI /
CLI.

This is **not** a security boundary - it's a clock check that any
mildly motivated user can defeat by rolling back the system date or
patching the constant. The point is just that a stray test build
handed to a tester doesn't quietly keep working forever after the test
window closes.

## Mechanics

In `ivi_installer/__main__.py`:

- `_BUILD_DATE = date(2026, 5, 8)` - bump this on every test build
  released to a third party.
- `_TRIAL_DAYS = 14`.
- `_trial_expired()` compares `date.today() - _BUILD_DATE` against
  `_TRIAL_DAYS`.
- GUI mode shows a `QMessageBox.critical` (Russian copy, mirroring the
  rest of the user-facing UI) before returning 2.
- CLI mode prints to stderr and returns 2.

The check fires **before** any device polling or worker thread
spin-up, so an expired build leaves no background state behind.

## Bumping for the next test build

When you ship the next build to a tester, edit `_BUILD_DATE` to that
day's date and bump the patch version. No other changes needed - the
14-day window is hardcoded for now.

Version: 0.22.1 → 0.22.2 (patch - small protection knob, no
behaviour change for anyone running a fresh build).
