# 0.9.0 - bundle adb.exe inside the Windows MSI

## Why

On Windows the previous flow downloaded `platform-tools-latest-windows.zip`
from `dl.google.com` on first launch (`adb.ensure_adb()` → `~/.ivi-installer/`).
That has two failure modes that bite at a customer site:

1. The download silently doesn't land - corporate firewall, no internet
   on the tech's laptop, the user dismisses the «download adb?» dialog.
   `find_adb()` then returns `None`, the GUI shows no devices, and the
   reason is buried in the log file.
2. The user manually drops `adb.exe` into PATH or alongside the app and
   it fails to run because `AdbWinApi.dll` / `AdbWinUsbApi.dll` aren't
   next to it (`STATUS_DLL_NOT_FOUND`, exit `0xC0000135`). Without the
   DLLs `adb devices` enumerates nothing.

## What changed

Ship the three Windows files inside the wheel:

```
ivi_installer/resources/platform-tools/windows/
  adb.exe              (8.5 MB)
  AdbWinApi.dll        (108 KB)
  AdbWinUsbApi.dll     ( 73 KB)
```

`platform-tools` v37.0.0 (build 14910828), pulled from
`https://dl.google.com/android/repository/platform-tools-latest-windows.zip`.

`adb.find_adb()` now checks a new "bundled" location (highest priority
after the explicit `ADB_BINARY` override) and resolves to a path
relative to this module - works for `pip install -e .` checkouts and
for the briefcase-built MSI alike (`app_packages/ivi_installer/...`).

macOS / Linux are unchanged: still no bundled adb, still rely on
`ensure_adb()` to fetch on first launch.

## Files touched

- `ivi_installer/__init__.py` - version 0.8.10 → 0.9.0
- `pyproject.toml` - version, force-include the three files, refresh
  the bundle comment
- `ivi_installer/adb.py` - new `_bundled_adb()` helper + reordered
  `find_adb()` lookup chain
- `ivi_installer/resources/platform-tools/windows/{adb.exe,AdbWinApi.dll,AdbWinUsbApi.dll}`

## MSI size impact

~+8.7 MB to the unpacked install dir (compresses to ~+3 MB inside the
MSI).

## What this still does NOT fix

If the head unit's USB driver isn't installed on Windows (no «ADB
Interface» entry in Device Manager), `adb devices -l` will still return
nothing. That requires the OEM USB driver - out of scope for this
change.
