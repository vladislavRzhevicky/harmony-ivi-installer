# 0.12.0 - pm-disable strategy: align with competitor recipe

The `pm_disable_install` strategy (the lighter, competitor-derived path)
stopped working on Deepal S09 with current firmware: the IVI showed the
"тип установки не поддерживается / not allowed" dialog on the driver
screen and rejected every install. Side-by-side analysis of our code vs.
the decompiled `AvatrAppInstaller` (where the same approach still works)
surfaced two first-order divergences:

1. We disabled `com.android.packageinstaller` on **user 0**, which is
   HEADLESS on Huawei IVI. The HwInstallPolicy hook actually renders its
   dialog from the active driver user (typically user 13 on Deepal). The
   disable on user 0 had zero effect on user 13's PackageInstaller, so
   the dialog kept firing.
2. We added `--user 0` (and `-t`) to `pm install` itself. Competitor
   passes neither - just `pm install -r -d -g -i <installer> <remote>`.
   On Huawei the `--user 0` form goes through stricter HEADLESS-only
   policy, which the disable above couldn't have neutralised anyway.

## Fix

In `_strategy_pm_disable_install` (`ivi_installer/strategies.py`):

- New helper `_resolve_packageinstaller_user(ctx, accum)` mirrors the
  competitor's `car_user_id` logic: parse `pm list users`, take
  `max(running non-zero ids)`, fall back to `min(non_zero)` then `0`.
  On a Deepal S09 with users 0/10/11/12/13 this lands on **13**.
- `pm disable-user --user 0` → `pm disable-user --user <pi_user>`.
- Removed the global `pm disable <pkg>` fallback. It was strictly more
  dangerous (kills PackageInstaller for every user simultaneously, not
  just the active driver) and on Huawei builds where per-user disable
  fails, the global form fails the same way. Competitor doesn't have it.
- On disable failure: best-effort `pm enable --user <pi_user>` + return
  `FAILED` with a clear message - the cascade then falls through to the
  HDB-broker strategy which is known to work on HarmonySpace 5.0.
- `pm install -r -t -d -g -i <installer> --user <seed_user> <remote>` →
  `pm install -r -d -g -i <installer> <remote>`. Dropped `-t` (no
  testOnly support - broker strategy still covers that via installFlags
  if anyone needs it) and dropped `--user` (let PMS install for
  USER_ALL by default, like the competitor does).
- Re-enable now symmetric: `pm enable --user <pi_user> <pkg>` instead of
  global `pm enable <pkg>`.
- Alt-installer retry now gated on a real `INSTALL_FAILED` code in the
  output (mirrors competitor's `'INSTALL_FAILED' in out` check), instead
  of any parse-shape failure.

## Untouched on purpose

- `_strategy_hdb_broker_install` - proven on HarmonySpace 5.0; bypasses
  PackageInstaller entirely via HDB session installFlags. Not affected
  by this change.
- `_resolve_targets` - broker strategy depends on its current
  `seed_user=0 + fan_out=screens` semantics. The new `pi_user` is a
  separate, locally-computed value used only for disable/enable.
- Stage 4 fan-out (`pm install-existing --user N`) and stage 5 runtime
  permission grants - these run after the install lands and don't go
  through the policy hook, so they don't need to change.

## Files

- `ivi_installer/strategies.py` - new `_resolve_packageinstaller_user`
  helper, rewrite of disable/install/enable block.
- `ivi_installer/__init__.py`, `pyproject.toml` - version 0.12.0.
