# 0.24.3 - Fix QThread crash on Windows when device is connected

## Symptom

On the Windows portable build, after a device is plugged in, the app
would crash within a few seconds - typically right when the user
clicked any UI element. Without a device, no crash. The Windows Event
Log recorded:

```
Faulting module: Qt6Core.dll
Exception code:  0xc0000409   (STATUS_STACK_BUFFER_OVERRUN)
Fault offset:    0x000000000001cf68
```

`0xc0000409` is the abort code Qt routes `qFatal()` through. The
faulting offset inside `Qt6Core.dll` is `QThread::~QThread()` -
exactly the same crash signature documented in the v0.22.1 fix:

> `QThread: Destroyed while thread is still running`

## Cause

Six side-worker kick functions in `MainWindow` followed the same
fire-and-forget pattern that v0.22.1 already identified as buggy in
the Store search path:

```python
worker = workers.UsersListWorker(self._selected_serial)   # local
thread = workers.run_in_thread(worker)                    # local
worker.finished.connect(lambda: self._drop_refs(thread, worker))
```

Once the function returns, the only Python references to `thread`
and `worker` are inside the lambda's closure, and the lambda is held
by Qt's connection table on `worker.finished`. That forms a cycle
(`worker → connection → lambda → worker`). Python's cyclic GC walks
that cycle on any allocation, including the ones triggered by Qt's
mouse-event dispatch - so a click during a still-running adb call
could finalise the QThread Python wrapper while the C++ thread was
mid-flight, hitting `QThread::~QThread()`'s qFatal.

The trigger was reliable on USB-connected devices because plugging in
fires two side workers immediately (`_kick_tz_read` +
`_kick_screen_categories_probe`), one of which (`UsersListWorker`)
runs `pm list users` + two `dumpsys` calls - long enough to overlap
with the next user click.

## Fix

Mirror the v0.22.1 fix - keep a non-circular strong reference in
`MainWindow._inflight: list[tuple]`, append `(thread, worker)` from
each affected kick, and remove from the list inside `_drop_refs`.

Affected sites (six fire-and-forget kicks):

- `_kick_pkg_list` - Tools-tab package list.
- `_on_bypass_refresh` - broker-status probe.
- `_kick_screen_categories_probe` - `pm list users` + display probe.
- `_kick_tz_read` - current-timezone read on connect / Timezone-tab open.
- `_kick_device_info_read` - Device-info-tab read.
- `_ensure_adb_present` - first-launch platform-tools download.

Active install paths (`_on_op_finished`) already kept refs in
`self._active_thread` / `self._active_worker` - those are unchanged.

`_on_refresh_imes` uses raw `threading.Thread`, not `QThread`, so it
is not affected.

## Tests

Existing test suite still passes - the race needs a real `QThread`
plus a real adb call in flight, which the unit tests mock out. No
new tests added (same rationale as v0.22.1).

## Version bump

0.24.2 → 0.24.3 (patch - fixes a hard crash users hit on the very
first device connect on Windows).
