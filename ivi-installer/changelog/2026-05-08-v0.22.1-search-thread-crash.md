# 0.22.1 - Fix QThread crash on rapid typing in Store search

v0.22.0 introduced a debounced type-ahead that kicks
`CompleteSearchWordWorker` after each pause in the user's typing.
The cancellation path replaced `self._search_suggest_worker` /
`_thread` references on every new keystroke, which dropped the last
Python reference to the previous QThread WHILE its SSL read was
still in flight. PySide6 then deallocated the QThreadWrapper, which
hits Qt's `QThread: Destroyed while thread is still running` qFatal
and aborts the process.

Crash signature (from the Apple crash report):

```
QtCore  QMessageLogger::fatal(...)
QtCore  QThread::~QThread()
QtCore  QThreadWrapper::~QThreadWrapper()
libshiboken6  SbkDeallocWrapperCommon
Python  _PyFrame_ClearExceptCode    ← Python GC running dealloc
```

## Fix

`MainWindow._on_store_search_suggestions_requested` no longer tries
to cancel the in-flight worker. Each request appends a `(thread,
worker)` tuple to `self._search_suggest_inflight`; the lambda closure
on `worker.finished` keeps both alive until the network call
completes, then `_on_search_suggest_finished` quits + waits + drops
the references.

Stale results are still filtered out - `StoreTab.set_search_suggestions`
compares the echoed keyword against the live input text and ignores
late arrivals for words the user has since cleared or replaced.

## Hot keywords

The hot-keyword path is unaffected - it has a `if self._hot_keywords_worker
is not None: return` guard up front (only one fetch ever fires per
session), so the same race can't happen there.

## Tests

309 tests still pass. No new tests added - the crash needs a real
QThread + a real network read in flight; the unit-test suite mocks
both away.

Version: 0.22.0 → 0.22.1 (patch - fix for a crash users would hit
on the very first burst of search input).
