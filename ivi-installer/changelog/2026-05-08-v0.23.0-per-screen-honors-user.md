# 0.23.0 - pm-disable strategy honours single-screen selection

## The bug

Reported by a tester after installing through the "pm-disable (lighter)"
strategy: with only **Driver** ticked in the per-screen grid, the APK
appeared on **all** active screens (driver, passenger, rear). The UI
contract was broken - picking one screen leaked installs onto every
multimedia user.

Root cause sat in stage 3 of `_strategy_pm_disable_install`. v0.12.0
mirrored the competitor's recipe verbatim:

```
pm install -r -d -g -i <installer> <remote>
```

Without `--user`, PMS treats the install as **USER_ALL** - i.e. mirror
the package to every Android user the firmware exposes. That happened
to be fine when v0.12 was written because we always wanted "all
multimedia screens", and `_resolve_targets` already routed multi-user
selections through that path. But the per-screen UI from 0.8.6 lets the
user narrow the target down to one screen, returning
`(seed_user=N, fan_out=[])` - and we ignored that on stage 3.

## Fix

`_strategy_pm_disable_install` stage 3 now branches on `fan_out`:

- **Single screen** (`fan_out` empty): `pm install --user <seed_user>` -
  install lands strictly on the picked user.
- **Multi screen / legacy default** (`fan_out` non-empty): keep the
  no-`--user` form so PMS does USER_ALL. pm-disable strategy has no
  per-user fan-out stage like the HDB-broker one, and USER_ALL covers
  every selected screen with one command. (Slightly imprecise when only
  some non-default subset was picked, but that's a separate refactor -
  see follow-up note below.)

Same branch applied to the alt-installer retry path.

## Second fix: `categorize_screens` fallback

While digging, surfaced a related UX bug on Deepal S09 (and likely
other Huawei IVI builds): `pm list users` returns opaque names like
`NoLoginUser`, `NoLoginUser_4`, `-1661017452`. None of these match the
heuristic in `_classify_user_name`, so every non-13 user fell into
`rear` and the **Passenger** bucket came back empty. The UI checkbox
went disabled / grey for that screen, even though Passenger physically
exists.

`categorize_screens` now detects "no name was decisive" and returns
the hardcoded `_FALLBACK_CATEGORIES` (driver=13, passenger=10/12,
rear=11) instead of force-fitting opaque names through default rules.
Heuristics still win whenever at least one name is recognisable.

## Files

- `ivi_installer/strategies.py` - stage 3 + alt retry now add
  `--user <seed_user>` when `fan_out` is empty.
- `ivi_installer/devices.py` - `categorize_screens` falls back to
  `_FALLBACK_CATEGORIES` if no user name maps to a known role.
- `tests/test_strategies.py`:
  - existing pm-disable test now asserts `--user 13` is present in
    the install command (was: asserted absent).
  - new test covers multi-screen → no `--user` (USER_ALL) path.
- `tests/test_devices.py` - four new cases for `categorize_screens`
  (named buckets, opaque-names fallback, no-multimedia fallback,
  partial naming keeps heuristics).
- `ivi_installer/__init__.py`, `pyproject.toml` - version 0.23.0.

## Follow-up (out of scope)

When the user picks a non-default *subset* of multiple screens
(e.g. Driver + Passenger but not Rear), pm-disable strategy still falls
into the USER_ALL path and the install reaches Rear too. Honest
multi-screen support there would need a per-user fan-out stage like
HDB-broker has - `pm install-existing --user N` for each picked id.
Tracked for a future patch; tester reported the all-three-screens
artefact only when picking exactly one screen, so the single-screen
fix is the immediate priority.
