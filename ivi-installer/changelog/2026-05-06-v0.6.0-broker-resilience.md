# 2026-05-06 - v0.6.0 Broker resilience

Bundle the AvatrHdbBroker daemon and auto-deploy it on demand. Removes
the implicit dependency on the leftover `app_process64 AvatrHdbBroker
38787` that's been running on the car since the 2026-04-28 workshop
session - installs now survive a car reboot or a `/data/local/tmp/`
wipe.

Pattern-setting feature: same shape (resource → strategy helper →
worker → Tools-tab section → unit tests) is reused for v0.7.0+ work
(force reinstall, IME enrolment).

## What changed

### Bundled resource

- New file `ivi_installer/resources/avatr-hdb-broker.jar` (~23 KB,
  pulled from `/data/local/tmp/avatr-hdb-broker.jar` on the car -
  same dexed bypass agent the workshop's Codex tool deployed).
- Wired into the wheel/.app build via
  `[tool.hatch.build.targets.wheel.force-include]` in
  `pyproject.toml`, mirroring how `hw-perm-grant.jar` is bundled.
- Verified post-build that the jar lands in
  `IVI Installer.app/Contents/Resources/app/ivi_installer/resources/`.

### `ivi_installer/strategies.py`

New constants:

- `HDB_BROKER_REMOTE = "/data/local/tmp/avatr-hdb-broker.jar"`
- `HDB_BROKER_LOG = "/data/local/tmp/avatr-hdb-broker.log"`
- `HDB_BROKER_RESOURCE = "avatr-hdb-broker.jar"`
- `HDB_BROKER_MAIN_CLASS = "AvatrHdbBroker"`

New helpers:

- `_broker_ping(timeout)` - boolean PING probe (no try/except in
  callers).
- `_load_broker_jar()` - read bundled jar bytes via
  `importlib.resources`, returns None if missing.
- `_md5_hex(blob)` - md5 hex digest helper.
- `_broker_remote_md5(ctx, accum)` - `md5sum` of the on-device jar
  (None if absent / unreadable).
- `_broker_pid(ctx, accum)` - parse `ps -A -o PID,ARGS` and pluck the
  PID of the `AvatrHdbBroker` line; smallest match wins.
- `_broker_uptime_s(ctx, accum, pid)` - read `ps -o ETIMES=` for
  process uptime in seconds.
- `_broker_launch_command()` - returns the shell incantation that
  starts the daemon as a detached process. Wrapped in `nohup sh -c "
  CLASSPATH=… app_process /system/bin AvatrHdbBroker 38787" >…log
  2>&1 &`. The `sh -c` wrap is intentional: plain `nohup … &` doesn't
  reliably detach on Huawei toybox, and `setsid`/`disown` aren't
  available. Log path matches the workshop-session broker so existing
  `tail -f` tooling keeps working.
- `_broker_deploy(ctx, accum)` - the heart of the feature:
  1. Load bundled jar; bail if missing from the bundle.
  2. Compare bundled md5 vs remote md5; push to
     `/data/local/tmp/` only if they differ (idempotent).
  3. Run launch command.
  4. Sleep 1.5 s, retry PING up to 3× with 1 s spacing.
  5. Returns True on PONG, False on giveup.

Updated:

- `_broker_health_check(ctx, accum)` - on PING failure it now calls
  `_broker_deploy` transparently (no user prompt - locked in
  `NEXT_SESSION.md`). Final fallback message is
  "broker won't start - check that the car is in dealer mode and
  adbd is alive".

New public API for the UI layer:

- `BrokerStatus` dataclass: `forwarded`, `alive`, `pid`, `uptime_s`,
  `port`, `remote_md5`, `bundled_md5`, `jar_matches`, plus
  `jar_present` derived property.
- `collect_broker_status(serial) -> BrokerStatus` - short-lived
  snapshot for the Tools tab. Failures collapse to `None`/`False`
  fields rather than raising.
- `redeploy_broker(serial, log) -> bool` - what the Tools-tab
  Redeploy button calls. Always re-runs the launch step even when
  PING currently succeeds, so the user has a reliable kick-it
  affordance.

Promoted strategy:

- `_strategy_app_process_helper` was a stub that returned SKIPPED.
  It now actually (re)deploys the broker via `_broker_deploy`.
- Descriptor relabelled "(Re)deploy AvatrHdbBroker daemon" with a
  description explaining when to use it. `applies=lambda _: False`
  preserved so the cascade never auto-runs it.

### `ivi_installer/ui/workers.py`

Two new QObject workers, both following the existing
`GrantRuntimePermsWorker` pattern (`_WorkerBase` ⇒ `log_line`,
`error`, `finished` + a `result` signal):

- `BrokerHealthWorker(serial)` → emits
  `result(serial, BrokerStatus)`. Wraps
  `strategies.collect_broker_status`.
- `BrokerRedeployWorker(serial)` → emits
  `result(serial, ok: bool)`. Wraps `strategies.redeploy_broker`.

### `ivi_installer/ui/main_window.py`

Tools tab: new "Bypass health" section appended under Diagnose,
separated by `_h_separator()`.

- Status label, monospaced font. Shows e.g.
  `✓ alive · pid=24354 · uptime=8d 2h · port=38787 · jar matches
  bundle` in green; on failure red with `✘ down · pid=- · jar
  absent · forwarded=true` etc. Format helper
  `_format_broker_status()` lives at module scope alongside existing
  `_format_tz_label()`. Companion `_format_uptime()` renders seconds
  as `s`, `m s`, `h m`, or `d h`.
- Refresh button - kicks `BrokerHealthWorker`, no busy-lock
  (read-only probe, can run alongside other ops).
- Redeploy broker button - kicks `BrokerRedeployWorker`, behind a
  `QMessageBox.question` confirm dialog (per locked decision: only
  the explicit button asks; auto-deploy in the install pipeline does
  NOT).

Gating: both buttons added to `_update_install_button()` so they
enable/disable in lockstep with the existing Tools-tab buttons.

After a successful redeploy, the UI auto-refreshes the status line
via `QTimer.singleShot(200, self._on_bypass_refresh)` so the new
PID/uptime show up without a manual click.

### `tests/test_strategies.py`

Adapted existing test:

- `test_app_process_helper_is_skipped` rewritten as
  `test_app_process_helper_redeploys_broker` - the strategy is no
  longer a stub. Mocks `_broker_deploy` so the test stays adb-free.
- `test_run_strategy_invokes_body_even_when_cascade_skips` rewritten
  to use the `diagnose` strategy as the "applies=False" example
  instead of `app_process_helper` (which now succeeds rather than
  SKIP-ing).

Seven new unit tests:

- `test_broker_jar_is_packaged` - resource is in the wheel and is a
  real DEX'd jar (PK header + `classes.dex` entry).
- `test_broker_launch_command_shape` - locks the launch incantation
  shape: `nohup sh -c "..." &`, CLASSPATH/main-class/port/log redirect
  inside the `sh -c` quotes (not outside), so detach actually works.
- `test_broker_deploy_skips_push_when_md5_matches` - md5 idempotency
  contract: matching remote md5 ⇒ no push call.
- `test_broker_deploy_pushes_when_md5_differs` - push lands at the
  canonical `/data/local/tmp/avatr-hdb-broker.jar` path.
- `test_broker_deploy_fails_when_ping_never_recovers` - PING never
  succeeding ⇒ deploy returns False.
- `test_broker_health_check_auto_deploys_on_ping_fail` - initial
  PING fails ⇒ `_broker_deploy` is called, post-deploy PING succeeds
  ⇒ health-check returns True.
- `test_broker_pid_parses_ps_output` - ps -A line picks the right
  PID, ignores the "grep AvatrHdbBroker" self-match.
- `test_collect_broker_status_assembles_snapshot` - UI smoke test:
  alive/pid/uptime/md5 fields populate as expected, `jar_matches`
  derives correctly.

Test count: **171 → 178 passed** (full suite, ~5 s).

## Locked decisions reused from `NEXT_SESSION.md`

- **Auto-redeploy without confirmation** when the install pipeline
  hits a missing broker. Only the explicit Tools → Redeploy button
  asks for confirmation.
- **In-process md5 idempotency** instead of always pushing - saves
  ~25 KB of adb push traffic per install on the happy path.

## Pending real-car verification (machine offline this session)

- That `nohup sh -c "..." &` actually detaches on Huawei toybox.
  Theory says yes (toybox sh implements job control via the wrapped
  shell), but no live host to confirm. If detach fails, the daemon
  dies when the adb shell session closes - easy to fix by switching
  to `setsid` or a tiny `daemonize` C helper if needed.
- Output shape of `ps -A -o PID,ARGS` on this firmware. If toybox
  pads columns differently, the PID parser will return None and the
  status panel shows `pid=-` even when the broker is alive
  (cosmetic; install path still works).
- That `md5sum /data/local/tmp/avatr-hdb-broker.jar` returns the
  standard `<hex>  <path>` two-column format on this firmware
  (tested in the prior session - should be fine).
