# 0.17.0 - F-Droid catalog is now opt-in

The Store tab no longer pulls the 50 MB F-Droid index on cold start.
By default it shows the curated extras (~4 entries today, growing
to ~60 in the next pass). A new "Show all F-Droid" toggle in the
toolbar adds the live F-Droid catalog on demand.

## Behaviour

- Cold start: extras-only store loaded from `resources/extras.json`,
  no network. Toggle is OFF.
- User clicks toggle ON: if `~/.ivi-installer/cache/catalog.sqlite3`
  exists, switch to it instantly. Else kick the background F-Droid
  fetch (50 MB index, gzipped on the wire, written into SQLite, then
  swapped into the tab). The previous-run cached SQLite is reused if
  fresh - no re-download.
- User clicks toggle OFF: switch back to the in-memory extras store.
  The on-disk SQLite stays warm so the next ON click is instant.
- Preference is persisted in `QSettings("deepal2", "ivi-installer")`
  under `store/fdroid_enabled`, so a user who enables the F-Droid
  catalog once gets the same view on every subsequent launch.

## Code

- `StoreTab` gains `fdroid_toggle` (a checkable QPushButton styled
  as an active-pill), `set_fdroid_toggle(checked)` to sync UI state
  without bouncing the signal, and a `fdroid_toggle_requested`
  signal carrying the new state.
- `StoreTab.set_catalog` no longer closes the previous store -
  `MainWindow` owns the lifecycle of `_extras_store` and `_full_store`
  and switches between them by reference. Two stores live side by
  side.
- `MainWindow._build_store_tab` reads `QSettings`, opens the cached
  SQLite if present, and seeds with extras otherwise. New
  `_on_store_fdroid_toggled` handles the swap (or kicks a fetch).
- `_on_catalog_fetch_result` updates `_full_store` and bumps the
  toggle so a successful fetch is reflected in the UI.

## Tests

Two new `test_store_tab.py` cases:
`test_fdroid_toggle_emits_signal_with_state` and
`test_set_fdroid_toggle_does_not_re_emit` (programmatic state sync
must not bounce the signal - that would loop the main_window
handler). Total 316 passed.
